\section{Results}
We report the first parsed topconf-4 live-model run on the held-out test split. The run covers 500 tasks, 3 agent protocols, and 2 DeepSeek-family models, for 3000 API trials over 3000 near-miss verification instances per model-agent aggregate. Raw JSONL traces are excluded from the repository by default; paper-ready aggregates are exported under \texttt{reports/stats\_topconf4\_parsed/}, \texttt{reports/failures\_topconf4\_parsed/}, \texttt{reports/costs\_topconf4\_parsed/}, and \texttt{reports/parse\_topconf4\_parsed/}.

The central pattern is that explicit specification construction changes the execution channel: spec-first is the only protocol that achieves non-zero task success in this run, even though direct and plan-first often produce syntactically valid DoneSpecs. This supports the DoneBench framing that completion-semantics construction, near-miss rejection, and execution under one's own criteria are separable capabilities rather than one undifferentiated ``agent success'' number.

DeepSeek Pro obtains higher criterion-completion F1 than Flash, especially under spec-first (68.0 vs. 56.7), but lower spec-first task success (8.4 vs. 25.4) and lower near-miss detection (44.3 vs. 50.6). We treat this as an important failure mode rather than a leaderboard claim: a model can articulate more gold-like criteria while producing brittle, overstrict, or poorly operationalized specs that hurt execution.

\subsection{Parse Transparency}

The official parsed run uses the compressed prompt and JSON repair path. Overall, 2692 of 3000 calls parsed cleanly (89.7\%); the remaining 308 rows used the deterministic fallback parser. Parse rate varies substantially by model and protocol, with DeepSeek Pro spec-first the main weak cell at 63.6\%. We therefore report parse transparency alongside model scores and avoid treating fallback-heavy cells as final model-capability estimates. The parse table is exported to \texttt{paper/tables/parse\_transparency.csv}.

\subsection{Cost and Runtime}

The parsed DeepSeek run used 3000 API calls, about 1.38M prompt tokens and 5.02M completion tokens, with an estimated API cost of \$3.56 under the configured price sheet. Summed provider latency was 124,141 seconds; with concurrent streaming execution, wall-clock time was roughly 33 minutes on the local run.
